This page documents the settings system in Zed, focusing on how settings are stored, layered from multiple sources, and merged to compute effective settings values. This covers the `SettingsStore` architecture, the layering precedence of different settings files, and how settings are resolved for specific contexts like file paths and programming languages.

For information about the visual settings editor UI, see [Settings UI](</zed-industries/zed/7.2-settings-ui>). For settings migration when configuration formats change, see [Settings Migration](</zed-industries/zed/7.3-settings-migration>). For keybindings specifically, see [Keymap System](</zed-industries/zed/7.4-keymap-system>).

## Architecture Overview

The settings system in Zed is built around the `SettingsStore`, a global singleton that manages all settings state and coordinates updates from multiple sources.

**SettingsStore Architecture**


Sources: [crates/settings/src/settings_store.rs141-159](<https://github.com/zed-industries/zed/blob/4109c9dd/crates/settings/src/settings_store.rs#L141-L159>) [crates/settings/src/settings_store.rs261-289](<https://github.com/zed-industries/zed/blob/4109c9dd/crates/settings/src/settings_store.rs#L261-L289>)

The `SettingsStore` maintains several key data structures:

Field| Type| Purpose  
---|---|---  
`default_settings`| `Rc<SettingsContent>`| Base settings from `assets/settings/default.json`  
`user_settings`| `Option<UserSettingsContent>`| User's global settings from `~/.config/zed/settings.json`  
`extension_settings`| `Option<Box<SettingsContent>>`| Settings provided by installed extensions  
`server_settings`| `Option<Box<SettingsContent>>`| Settings from remote server (SSH/collaborative)  
`global_settings`| `Option<Box<SettingsContent>>`| Release channel and profile overrides  
`merged_settings`| `Rc<SettingsContent>`| Final merged settings from all sources  
`local_settings`| `BTreeMap<(WorktreeId, Arc<RelPath>), SettingsContent>`| Per-directory project settings  
`setting_values`| `HashMap<TypeId, Box<dyn AnySettingValue>>`| Type-erased storage for each `Settings` implementation  
  
Sources: [crates/settings/src/settings_store.rs141-159](<https://github.com/zed-industries/zed/blob/4109c9dd/crates/settings/src/settings_store.rs#L141-L159>)

## Settings Sources and Precedence

Settings in Zed are layered from multiple sources with a well-defined precedence order. Settings from higher-precedence sources override those from lower-precedence sources.

**Settings Precedence Order (lowest to highest)**

### Default Settings

The base layer is `default.json`, which provides sensible defaults for all settings. This file is embedded in the Zed binary and serves as documentation for available settings.

Sources: [assets/settings/default.json1-951](<https://github.com/zed-industries/zed/blob/4109c9dd/assets/settings/default.json#L1-L951>) [crates/settings/src/settings_store.rs262-265](<https://github.com/zed-industries/zed/blob/4109c9dd/crates/settings/src/settings_store.rs#L262-L265>)

### User Settings

User settings are stored at `~/.config/zed/settings.json` (or `$XDG_CONFIG_HOME/zed/settings.json` on Linux). These override defaults and apply globally to all projects.


Sources: [docs/src/configuring-zed.md14-27](<https://github.com/zed-industries/zed/blob/4109c9dd/docs/src/configuring-zed.md#L14-L27>)

### Release Channel Overrides

Within user settings, you can scope overrides to specific release channels using top-level keys:


When running the nightly build, the effective theme would be "One Dark" and vim mode would be enabled.

### Settings Profiles

Settings profiles allow temporarily applying a set of settings on top of your base configuration:


Profiles are activated through the `ActiveSettingsProfileName` global and automatically merged into the settings stack.

Sources: [crates/settings/src/settings_store.rs477-491](<https://github.com/zed-industries/zed/blob/4109c9dd/crates/settings/src/settings_store.rs#L477-L491>) [crates/settings/src/settings_store.rs291-297](<https://github.com/zed-industries/zed/blob/4109c9dd/crates/settings/src/settings_store.rs#L291-L297>)

### Project Settings

Project-specific settings are stored in `.zed/settings.json` at the root of a project directory. These override user settings for files within that project:


You can create multiple `.zed/settings.json` files in subdirectories for more granular control. When resolving settings for a specific file, Zed walks up the directory tree and applies all applicable project settings in order.

Sources: [docs/src/configuring-zed.md36-43](<https://github.com/zed-industries/zed/blob/4109c9dd/docs/src/configuring-zed.md#L36-L43>) [crates/settings/src/settings_store.rs152](<https://github.com/zed-industries/zed/blob/4109c9dd/crates/settings/src/settings_store.rs#L152-L152>)

### EditorConfig Integration

`.editorconfig` files provide the highest-precedence settings layer. These files follow the [EditorConfig](<https://editorconfig.org/>) standard and override specific text-editing settings:


EditorConfig properties are parsed using the `ec4rs` library and merged into `LanguageSettings` for the specific file being edited.

Sources: [crates/language/src/language_settings.rs475-512](<https://github.com/zed-industries/zed/blob/4109c9dd/crates/language/src/language_settings.rs#L475-L512>) [crates/settings/src/settings_store.rs153](<https://github.com/zed-industries/zed/blob/4109c9dd/crates/settings/src/settings_store.rs#L153-L153>)

## Settings Merging

Settings from different sources are combined using the `MergeFrom` trait, which defines how to merge a source settings object into a destination.

**Merge Flow**


Sources: [crates/settings/src/settings_store.rs492-580](<https://github.com/zed-industries/zed/blob/4109c9dd/crates/settings/src/settings_store.rs#L492-L580>)

### The MergeFrom Trait

The `MergeFrom` trait defines how settings are merged:


The merge semantics depend on the field type:

Type| Merge Behavior  
---|---  
`Option<T>`| If `other` is `Some`, replace with `other`'s value  
Scalar values| Replace with `other`'s value if present  
`HashMap<K, V>`| Recursively merge entries; `other` entries override  
`Vec<T>`| Replace entirely with `other`'s vec  
Nested structs| Recursively merge fields  
  
The merge is performed in precedence order, so higher-precedence sources always win.

Sources: [crates/settings/src/merge_from.rs1-102](<https://github.com/zed-industries/zed/blob/4109c9dd/crates/settings/src/merge_from.rs#L1-L102>)

### Language-Specific Settings Merging

Language-specific settings require special merging logic because they exist in a nested structure:


The merge algorithm:

  1. User's global defaults override default global defaults
  2. User's global defaults override default language-specific settings
  3. User's language-specific settings override default language-specific settings


This ensures that `{ "tab_size": 2 }` in user settings applies to all languages unless explicitly overridden in a language-specific block.

Sources: [crates/settings/src/settings_content/language.rs30-56](<https://github.com/zed-industries/zed/blob/4109c9dd/crates/settings/src/settings_content/language.rs#L30-L56>)

## Registering New Settings

To add a new type of setting to Zed:

  1. Define your settings struct implementing the `Settings` trait
  2. Use the `RegisterSetting` derive macro
  3. Implement `from_settings` to extract values from `SettingsContent`
  4. Call `register()` during initialization


The `RegisterSetting` macro generates the necessary inventory registration so the settings are automatically loaded.

Sources: [crates/settings/src/settings_store.rs44-125](<https://github.com/zed-industries/zed/blob/4109c9dd/crates/settings/src/settings_store.rs#L44-L125>) [crates/editor/src/editor_settings.rs16-60](<https://github.com/zed-industries/zed/blob/4109c9dd/crates/editor/src/editor_settings.rs#L16-L60>)

### File Watching

The `SettingsStore` watches settings files for changes using the `watch_config_file` utility:


This returns a stream of file contents whenever the file changes. The store parses the updated JSON and triggers a recomputation.

Sources: [crates/settings/src/settings_file.rs1-93](<https://github.com/zed-industries/zed/blob/4109c9dd/crates/settings/src/settings_file.rs#L1-L93>)

### Recomputing Values

The `recompute_values` method rebuilds the merged settings and updates all registered setting types:

  1. Merge all settings sources into `merged_settings`
  2. For each registered `Settings` type, call `from_settings(&merged_settings)`
  3. Update the corresponding `SettingValue` in the `setting_values` map
  4. Notify GPUI to trigger re-renders


Sources: [crates/settings/src/settings_store.rs492-580](<https://github.com/zed-industries/zed/blob/4109c9dd/crates/settings/src/settings_store.rs#L492-L580>)

## Local Settings Storage

Project-specific settings are stored in a `BTreeMap` keyed by `(WorktreeId, Arc<RelPath>)`:


The ordering of the `BTreeMap` ensures that when looking up settings for a file path, Zed can efficiently find all applicable settings by iterating in reverse order (from most specific to least specific path).

**Local Settings Lookup**

When settings formats change between Zed versions, the migration system automatically updates user configuration files. This is handled by the `migrator` crate, which is covered in detail in [Settings Migration](</zed-industries/zed/7.3-settings-migration>).

## Error Handling

Settings parsing errors are tracked in `SettingsStore.file_errors`:


When a settings file has parse errors, Zed continues using the last known-good configuration and displays an error to the user. The store tracks which specific file has errors to provide helpful error messages.
