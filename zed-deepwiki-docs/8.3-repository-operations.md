This page documents the git repository operations performed in Zed, including staging, committing, branching, pushing, pulling, and stashing. These operations are exposed through both the Git Panel UI and programmatic interfaces.

For information about the Git Panel UI and user interactions, see [Git Panel and UI](</zed-industries/zed/8.1-git-panel-and-ui>). For details about repository state management and event handling, see [Git Store and State Management](</zed-industries/zed/8.2-git-store-and-state-management>). For diff computation and visualization, see [Diff System](</zed-industries/zed/8.4-diff-system>).

## Architecture Overview

Repository operations in Zed follow a layered architecture that separates UI concerns from git implementation details and handles both local and remote (SSH) execution contexts.


**Sources:** [crates/git_ui/src/git_panel.rs560-601](<https://github.com/zed-industries/zed/blob/4109c9dd/crates/git_ui/src/git_panel.rs#L560-L601>) [crates/project/src/git_store.rs278-293](<https://github.com/zed-industries/zed/blob/4109c9dd/crates/project/src/git_store.rs#L278-L293>) [crates/git/src/repository.rs657-712](<https://github.com/zed-industries/zed/blob/4109c9dd/crates/git/src/repository.rs#L657-L712>)

## Staging Operations

Staging operations modify the git index to prepare changes for commit. These operations can target individual files, hunks, or entire changesets.

### Stage/Unstage Paths

The fundamental staging operation works with a list of repository-relative paths.


**Sources:** [crates/project/src/git_store.rs2064-2143](<https://github.com/zed-industries/zed/blob/4109c9dd/crates/project/src/git_store.rs#L2064-L2143>) [crates/git/src/repository.rs514-527](<https://github.com/zed-industries/zed/blob/4109c9dd/crates/git/src/repository.rs#L514-L527>)

The staging operation:

  1. Reads current file contents from the worktree
  2. Writes contents to the index for specified paths
  3. Removes index entries for deleted files
  4. Updates repository status after completion


**Sources:** [crates/git/src/repository.rs514-527](<https://github.com/zed-industries/zed/blob/4109c9dd/crates/git/src/repository.rs#L514-L527>) [crates/git/src/repository.rs1460-1558](<https://github.com/zed-industries/zed/blob/4109c9dd/crates/git/src/repository.rs#L1460-L1558>)

### Bulk Staging

The Git Panel implements efficient bulk staging for contiguous file selections using an optimization called "bulk staging".

Operation| Description| Implementation  
---|---|---  
`stage_all`| Stage all unstaged changes| Iterates all unstaged entries  
`unstage_all`| Unstage all staged changes| Iterates all staged entries  
`stage_range`| Stage entries between anchor and cursor| Uses `BulkStaging` state  
Hunk staging| Stage individual diff hunks| Calls `set_index_text` directly  
  
**Sources:** [crates/git_ui/src/git_panel.rs1423-1572](<https://github.com/zed-industries/zed/blob/4109c9dd/crates/git_ui/src/git_panel.rs#L1423-L1572>)

The bulk staging mechanism tracks a `BulkStaging` struct containing the repository ID and anchor path. When staging with Shift held, it stages all paths between the anchor and cursor in tree order.

**Sources:** [crates/git_ui/src/git_panel.rs603-607](<https://github.com/zed-industries/zed/blob/4109c9dd/crates/git_ui/src/git_panel.rs#L603-L607>) [crates/git_ui/src/git_panel.rs1500-1572](<https://github.com/zed-industries/zed/blob/4109c9dd/crates/git_ui/src/git_panel.rs#L1500-L1572>)

## Commit Operations

Commit operations create new git commits from staged changes, with support for amending, sign-off, and custom author information.

### Commit Flow


**Sources:** [crates/git_ui/src/commit_modal.rs110-227](<https://github.com/zed-industries/zed/blob/4109c9dd/crates/git_ui/src/commit_modal.rs#L110-L227>) [crates/project/src/git_store.rs2144-2233](<https://github.com/zed-industries/zed/blob/4109c9dd/crates/project/src/git_store.rs#L2144-L2233>)

### Commit Options

The commit operation accepts several options through the `CommitOptions` struct:

Option| Type| Description  
---|---|---  
`amend`| `bool`| Replace the previous commit instead of creating new one  
`signoff`| `bool`| Add "Signed-off-by" line to commit message  
`name_and_email`| `Option<(String, String)>`| Override committer identity  
  
**Sources:** [crates/git/src/repository.rs150-153](<https://github.com/zed-industries/zed/blob/4109c9dd/crates/git/src/repository.rs#L150-L153>) [crates/proto/proto/git.proto386-400](<https://github.com/zed-industries/zed/blob/4109c9dd/crates/proto/proto/git.proto#L386-L400>)

### Amend Workflow

When amending a commit, the Git Panel loads the previous commit message and replaces it when the new commit is created.


**Sources:** [crates/git_ui/src/git_panel.rs1702-1731](<https://github.com/zed-industries/zed/blob/4109c9dd/crates/git_ui/src/git_panel.rs#L1702-L1731>) [crates/git_ui/src/commit_modal.rs131-150](<https://github.com/zed-industries/zed/blob/4109c9dd/crates/git_ui/src/commit_modal.rs#L131-L150>)

### Git Hooks

Before committing, Zed executes the `pre-commit` hook if it exists. The hook runs with the repository's environment and can prevent the commit if it exits with a non-zero status.

**Sources:** [crates/git/src/repository.rs528-533](<https://github.com/zed-industries/zed/blob/4109c9dd/crates/git/src/repository.rs#L528-L533>) [crates/git/src/repository.rs1691-1720](<https://github.com/zed-industries/zed/blob/4109c9dd/crates/git/src/repository.rs#L1691-L1720>)

## Remote Operations

Remote operations synchronize local commits with remote repositories through push, pull, and fetch operations.

### Push Operation

Push sends local commits to a remote repository with optional force and set-upstream flags.


**Sources:** [crates/git_ui/src/git_panel.rs1880-2028](<https://github.com/zed-industries/zed/blob/4109c9dd/crates/git_ui/src/git_panel.rs#L1880-L2028>) [crates/git/src/repository.rs567-578](<https://github.com/zed-industries/zed/blob/4109c9dd/crates/git/src/repository.rs#L567-L578>)

### Push Options

Option| Proto Value| Description  
---|---|---  
Normal| None| Standard push  
`SetUpstream`| `0`| Sets upstream tracking branch  
`Force`| `1`| Force push, overwriting remote history

## Stash Operations

Stash operations save and restore uncommitted changes, useful for temporarily switching context.

### Stash Structure


**Sources:** [crates/git/src/stash.rs1-69](<https://github.com/zed-industries/zed/blob/4109c9dd/crates/git/src/stash.rs#L1-L69>) [crates/proto/proto/git.proto339-345](<https://github.com/zed-industries/zed/blob/4109c9dd/crates/proto/proto/git.proto#L339-L345>)

### Stash Operations

Operation| Description| Behavior  
---|---|---  
`stash_paths`| Create stash from paths| Saves working directory changes for specified paths  
`stash_pop`| Apply and remove latest stash| Applies stash at index and removes it from list  
`stash_apply`| Apply stash without removing| Applies stash at index but keeps it in list  
`stash_drop`| Remove stash entry| Deletes stash at index without applying  
`stash_entries`| List all stashes| Returns all stash entries with metadata

## Reset and Restore Operations

Reset and restore operations modify repository state and working directory contents.

### Reset Modes

Git reset supports two modes in Zed:

Mode| Effect on Index| Effect on Worktree| Use Case  
---|---|---|---  
`Soft`| Unchanged| Unchanged| Uncommit but keep changes staged  
`Mixed`| Reset| Unchanged| Uncommit and unstage changes
