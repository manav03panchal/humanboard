The Keymap System manages the mapping of keystroke sequences to actions in Zed. It loads keymap definitions from JSON files, validates bindings, supports context-aware key matching, and provides mechanisms for user customization and migration of legacy keymap formats.

For information about the action dispatch mechanism that uses these keymaps, see the Event Flow and Input Handling documentation in [2.4](</zed-industries/zed/2.4-event-flow-and-input-handling>). For settings configuration including the `base_keymap` setting, see [7.1](</zed-industries/zed/7.1-settings-store-and-layering>).

## Keymap File Structure

Keymap files consist of an array of `KeymapSection` objects, each defining a set of key bindings active in specific contexts.

### KeymapSection Components


**File Format Example:**


The `KeymapFile` struct wraps a vector of `KeymapSection` objects. Each section contains:

  * `context`: A string predicate determining when bindings are active
  * `use_key_equivalents`: Platform-specific key position mapping (macOS only)
  * `bindings`: Map of keystroke sequences to actions
  * `unrecognized_fields`: Captures unknown fields for validation

## Context Predicates

Context predicates determine when key bindings are active. They use a boolean expression syntax evaluated against the current UI state.

### Context Evaluation


### Context Syntax

Context predicates support:

  * **Simple names** : `"Editor"`, `"Workspace"`
  * **Boolean operators** : `"Editor && mode == full"`, `"Pane || Menu"`
  * **Negation** : `"!showing_completions"`
  * **Equality checks** : `"extension == md"`, `"mode == auto_height"`
  * **Platform checks** : Built into context evaluation


**Common Context Examples:**

Context| When Active  
---|---  
`"Editor"`| Any editor is focused  
`"Editor && mode == full"`| Full-height editor is focused  
`"Pane"`| Pane container is focused  
`"BufferSearchBar"`| Buffer search UI is open  
`"Editor && !showing_completions"`| Editor focused without completion menu  
`"Editor && extension == md"`| Markdown file in editor

## Keystroke Sequences

Keystroke sequences define multi-key shortcuts. Each keystroke consists of modifiers and a key, and sequences are space-separated.

### Keystroke Syntax
    
    
    <modifiers>-<key> <modifiers>-<key> ...
    

**Modifier Symbols:**

  * `cmd`: Command key (macOS) / Windows key
  * `ctrl`: Control key
  * `alt`: Alt/Option key
  * `shift`: Shift key
  * `fn`: Function key (macOS)
  * `super`, `win`: Platform super keys


**Examples:**

Keystroke String| Description  
---|---  
`"cmd-s"`| Command+S (single keystroke)  
`"cmd-k cmd-b"`| Command+K followed by Command+B (sequence)  
`"ctrl-shift-p"`| Control+Shift+P  
`"cmd-k s"`| Command+K followed by S (no modifier on second)

## Action Binding

Actions are bound to keystroke sequences through the keymap file. At runtime, the binding system constructs `KeyBinding` objects that link keystrokes, context predicates, and action instances.

### Binding Construction


### Action Building Process

When a keymap binding is loaded:

  1. **Action Name Extraction** : The action string (e.g., `"workspace::Save"`) is parsed
  2. **Registry Lookup** : The action name is looked up in the GPUI action registry using `register_action!` mappings
  3. **Argument Deserialization** : If arguments exist, they're deserialized into the action's parameter struct
  4. **Action Construction** : The action builder creates a `Box<dyn Action>` instance
  5. **Error Handling** : If action is unknown or args are invalid, produces `ActionBuildError`


**Sources:** [crates/settings/src/keymap_file.rs183-260](<https://github.com/zed-industries/zed/blob/4109c9dd/crates/settings/src/keymap_file.rs#L183-L260>) [crates/settings/src/keymap_file.rs26-44](<https://github.com/zed-industries/zed/blob/4109c9dd/crates/settings/src/keymap_file.rs#L26-L44>)

## User Customization

Users customize keybindings by creating or editing `~/.config/zed/keymap.json`. User bindings take precedence over base keymap bindings.

### Precedence Rules

Keymap bindings are evaluated in order of precedence:


### Within Same File Precedence

Within a single keymap file, later bindings override earlier ones for the same keystroke at the same context depth:

  1. **Last Binding Wins** : If two sections at the same context specificity bind the same keystroke, the later one is used for execution
  2. **First Binding for Display** : For UI display purposes (showing keyboard shortcuts), the first binding for an action is preferred

## Key Equivalents

The `use_key_equivalents` feature enables QWERTY-based key positioning for non-QWERTY keyboard layouts. This is currently only supported on macOS.

### Key Equivalents System


When enabled, keys are matched based on their physical position on a QWERTY layout rather than their character output. This allows:

  * Muscle memory preservation across keyboard layouts
  * Consistent positioning for common commands
  * Support for international keyboards

## Integration with Event Dispatch

The keymap system integrates with the GPUI event dispatch mechanism described in [2.4](</zed-industries/zed/2.4-event-flow-and-input-handling>).

### Keystroke to Action Flow


### KeyBinding Dispatch

When the `Keymap` finds a matching binding:

  1. **Context Evaluation** : The binding's `KeyBindingContextPredicate` is evaluated against the current `KeyContext`
  2. **Keystroke Matching** : The keystroke sequence is compared against pending keystrokes (for multi-keystroke sequences)
  3. **Action Dispatch** : The bound `Action` is dispatched through the `DispatchTree`
  4. **Two-Phase Handling** : Actions are first offered in capture phase (root to leaf), then bubble phase (leaf to root)


The `KeyBinding` struct contains:

  * `keystrokes: Vec<KeybindingKeystroke>` \- The keystroke sequence
  * `action: Box<dyn Action>` \- The action instance to execute
  * `context_predicate: KeyBindingContextPredicate` \- When the binding is active
