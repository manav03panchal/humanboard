The Settings UI provides a visual interface for browsing and editing Zed's configuration settings. It presents a searchable, categorized view of settings with appropriate controls (toggles, dropdowns, text fields, etc.) for each setting type, reading from and writing to the underlying settings files.

For information about how settings are stored, merged, and resolved across multiple layers (default, user, project), see [Settings Store and Layering](</zed-industries/zed/7.1-settings-store-and-layering>). For details on automatic migration of old settings formats, see [Settings Migration](</zed-industries/zed/7.3-settings-migration>).

## Overview

The Settings UI is implemented as a standalone window (`SettingsWindow`) that can be opened via the `OpenSettings` action. It allows users to:

  * Browse settings organized into hierarchical pages
  * Search settings by name, description, or JSON path
  * Toggle between user settings and project-specific settings
  * Edit settings with type-appropriate controls
  * Reset settings to their default values
  * Open the underlying JSON file for manual editing


The system bridges between user-friendly UI controls and the structured JSON settings files managed by `SettingsStore`.

## Architecture

### SettingsWindow Structure


The `SettingsWindow` struct contains the main state:

  * `pages: Vec<SettingsPage>` \- All setting pages loaded from `page_data.rs`
  * `current_file: SettingsUiFile` \- Whether editing user or project settings
  * `files: Vec<(SettingsUiFile, FocusHandle)>` \- Available files to edit
  * `navbar_entries: Vec<NavBarEntry>` \- Flattened navigation structure
  * `search_bar: Entity<Editor>` \- Search input field
  * `filter_table: Vec<Vec<bool>>` \- Visibility state for search/file filtering


Sources: [crates/settings_ui/src/settings_ui.rs665-693](<https://github.com/zed-industries/zed/blob/4109c9dd/crates/settings_ui/src/settings_ui.rs#L665-L693>)

### Settings Page Data Model


Each `SettingItem` contains:

  * `title` and `description` \- Display text
  * `field: Box<dyn AnySettingField>` \- Type-erased accessor
  * `files: SettingsFiles` \- Whether editable in user/project files
  * `metadata: Option<Box<SettingsFieldMetadata>>` \- Rendering hints


The `SettingField<T>` provides type-safe access:

  * `pick: fn(&SettingsContent) -> Option<&T>` \- Read current value
  * `write: fn(&mut SettingsContent, Option<T>)` \- Update value
  * `json_path: Option<&'static str>` \- JSON key path for deep linking

## Rendering System

### Field Renderer Architecture


The rendering system uses a type-indexed registry to dispatch to appropriate renderers:

  1. During initialization (`init_renderers`), renderers are registered for each setting type
  2. At render time, the `TypeId` of the setting's type is used to look up the renderer
  3. The `AnySettingField` is downcast to the concrete `SettingField<T>`
  4. The type-specific renderer creates the appropriate control


Common renderer types:

  * **Toggle button** \- `Switch` component for boolean values
  * **Text field** \- `SettingsInputField` for string/number input
  * **Dropdown** \- `EnumVariantDropdown` for enum variants
  * **Font picker** \- Modal for selecting fonts with preview
  * **Theme picker** \- Modal for selecting themes with preview


Sources: [crates/settings_ui/src/settings_ui.rs231-332](<https://github.com/zed-industries/zed/blob/4109c9dd/crates/settings_ui/src/settings_ui.rs#L231-L332>) [crates/settings_ui/src/settings_ui.rs404-518](<https://github.com/zed-industries/zed/blob/4109c9dd/crates/settings_ui/src/settings_ui.rs#L404-L518>)

### Control Rendering Example

For a boolean setting:


The control creation follows this pattern:

  1. Read current value via `field.pick(settings_content)`
  2. Create interactive control (e.g., `Switch::new(id, checked)`)
  3. Attach `on_click` handler that calls `update_settings_file`
  4. Handler uses `field.write(settings_content, new_value)` to update

## Data Flow

### Reading Settings


When rendering a setting:

  1. `SettingsStore::get_content_for_file(file)` returns the appropriate `SettingsContent`
  2. `field.pick(settings_content)` extracts the specific value
  3. The value is passed to the control renderer
  4. For indicating which file sets the value, `field.file_set_in(file, cx)` is used


Sources: [crates/settings_ui/src/settings_ui.rs162-224](<https://github.com/zed-industries/zed/blob/4109c9dd/crates/settings_ui/src/settings_ui.rs#L162-L224>) [crates/settings_ui/src/settings_ui.rs782-848](<https://github.com/zed-industries/zed/blob/4109c9dd/crates/settings_ui/src/settings_ui.rs#L782-L848>)

### Writing Settings


The write flow for a setting change:

  1. User interacts with control (clicks toggle, enters text, selects dropdown option)
  2. Event handler calls `update_settings_file(file, None, cx, callback)`
  3. Callback receives mutable `SettingsContent` and worktree_id
  4. Uses `field.write(settings, Some(new_value))` to update
  5. `update_value_in_json_text` preserves formatting while updating JSON
  6. File is written, triggering `SettingsStore` reload
  7. UI re-renders with new values


Sources: [crates/settings_ui/src/settings_ui.rs1366-1484](<https://github.com/zed-industries/zed/blob/4109c9dd/crates/settings_ui/src/settings_ui.rs#L1366-L1484>) [crates/settings_ui/src/settings_ui.rs1494-1629](<https://github.com/zed-industries/zed/blob/4109c9dd/crates/settings_ui/src/settings_ui.rs#L1494-L1629>)

## Search Functionality

### Search Index Architecture


The search system maintains three parallel data structures:

  1. **BM25 Engine** \- Full-text search with TF-IDF ranking for natural language queries
  2. **Fuzzy Candidates** \- Character-level fuzzy matching for substring queries
  3. **Lookup Table** \- Maps search result indices to `(page_index, header_index, item_index)`


When searching:

  * Text queries use BM25 ranking
  * Queries starting with `#` use JSON path matching
  * Results update the `filter_table` to show/hide matching settings
  * Navigation entries automatically expand to show matching items


Sources: [crates/settings_ui/src/settings_ui.rs695-706](<https://github.com/zed-industries/zed/blob/4109c9dd/crates/settings_ui/src/settings_ui.rs#L695-L706>) [crates/settings_ui/src/settings_ui.rs1724-1843](<https://github.com/zed-industries/zed/blob/4109c9dd/crates/settings_ui/src/settings_ui.rs#L1724-L1843>)

### Search Query Processing


Query types:

  * **Empty query** \- Show all items for current file scope
  * **Text query** \- BM25 search on title/description
  * **JSON path query** (`#setting.key.path`) - Match JSON paths
  * **Combined** \- Both BM25 and fuzzy results merged

## File Scope Management

### File Selection and Filtering


Each setting item specifies which file types it can appear in:

  * `USER` \- Editable in user settings (`~/.config/zed/settings.json`)
  * `PROJECT` \- Editable in project settings (`.zed/settings.json`)
  * `USER | PROJECT` \- Editable in both


When switching files:

  1. `change_file(index, window, cx)` updates `current_file`
  2. `recompute_filter_table()` updates visibility for all items
  3. Items with incompatible flags are hidden
  4. Navigation rebuilds to only show accessible sections


Sources: [crates/settings_ui/src/settings_ui.rs1979-2068](<https://github.com/zed-industries/zed/blob/4109c9dd/crates/settings_ui/src/settings_ui.rs#L1979-L2068>) [crates/settings_ui/src/settings_ui.rs2158-2227](<https://github.com/zed-industries/zed/blob/4109c9dd/crates/settings_ui/src/settings_ui.rs#L2158-L2227>) [crates/settings_ui/src/page_data.rs7-9](<https://github.com/zed-industries/zed/blob/4109c9dd/crates/settings_ui/src/page_data.rs#L7-L9>)

### Multiple Project Files


The settings window can edit multiple project files simultaneously:

  1. On initialization, `fetch_files()` discovers all worktrees
  2. For each worktree, checks for `.zed/settings.json`
  3. Creates a `SettingsUiFile::Project` entry for each found file
  4. User can switch between files using the tab selector
  5. Each file maintains independent undo/redo history


Sources: [crates/settings_ui/src/settings_ui.rs2229-2284](<https://github.com/zed-industries/zed/blob/4109c9dd/crates/settings_ui/src/settings_ui.rs#L2229-L2284>) [crates/settings_ui/src/settings_ui.rs2570-2629](<https://github.com/zed-industries/zed/blob/4109c9dd/crates/settings_ui/src/settings_ui.rs#L2570-L2629>)

## Integration with SettingsStore

### Reading Values from Store

The Settings UI queries `SettingsStore` to get current values and determine which file sets each value:


Key methods:

  * `get_content_for_file(file: SettingsFile)` \- Returns merged settings for a specific file
  * `get_value_from_file(file, pick_fn)` \- Finds which file actually defines a value
  * `get_value_up_to_file(file, pick_fn)` \- Gets effective value up to a specific precedence level


These enable the UI to:

  * Display current values
  * Show which file sets each value
  * Determine if a value can be reset (differs from default)
  * Highlight overridden settings


Sources: [crates/settings_ui/src/settings_ui.rs175-179](<https://github.com/zed-industries/zed/blob/4109c9dd/crates/settings_ui/src/settings_ui.rs#L175-L179>) [crates/settings/src/settings_store.rs363-394](<https://github.com/zed-industries/zed/blob/4109c9dd/crates/settings/src/settings_store.rs#L363-L394>)

### Writing Values to Store

Updates flow through the JSON manipulation layer to preserve formatting:


The `update_settings_file` function:

  1. Loads current JSON text from file
  2. Parses to `SettingsContent` struct
  3. Applies modifications via callback
  4. Computes minimal JSON edits to preserve formatting
  5. Writes updated JSON back to file
  6. `SettingsStore` detects file change and reloads
  7. UI receives notification and re-renders


This preserves:

  * Comment lines
  * Indentation style
  * Key ordering
  * Formatting of unchanged values


Sources: [crates/settings_ui/src/settings_ui.rs1494-1587](<https://github.com/zed-industries/zed/blob/4109c9dd/crates/settings_ui/src/settings_ui.rs#L1494-L1587>) [crates/settings/src/settings_store.rs832-955](<https://github.com/zed-industries/zed/blob/4109c9dd/crates/settings/src/settings_store.rs#L832-L955>)

## Dynamic Settings

### Dynamic Item Pattern

Some settings have a discriminant that changes the available sub-fields. For example, theme selection can be either static (one theme) or dynamic (light/dark themes):


The pattern allows:

  1. **Discriminant setting** \- Determines which variant is active (enum dropdown)
  2. **Pick function** \- Returns current variant index
  3. **Field sets** \- One `Vec<SettingItem>` per variant
  4. **Rendering** \- Shows discriminant + fields for active variant


Example usage for theme settings:

  * Discriminant: "Static" vs "Dynamic" mode selector
  * Static fields: Single theme name picker
  * Dynamic fields: Mode dropdown, light theme picker, dark theme picker


Sources: [crates/settings_ui/src/settings_ui.rs728-755](<https://github.com/zed-industries/zed/blob/4109c9dd/crates/settings_ui/src/settings_ui.rs#L728-L755>) [crates/settings_ui/src/page_data.rs274-429](<https://github.com/zed-industries/zed/blob/4109c9dd/crates/settings_ui/src/page_data.rs#L274-L429>)

## Navigation and Keyboard Control

### Navigation Structure


The navigation system:

  1. Flattens hierarchical pages into a linear list (`navbar_entries`)
  2. Assigns focus handles for keyboard navigation
  3. Tracks expanded/collapsed state per entry
  4. Manages tab order: navbar (0-1) → header (2-3) → content (4-5)


Keyboard actions:

  * `ToggleFocusNav` \- Switch between sidebar and content
  * `ExpandNavEntry` / `CollapseNavEntry` \- Tree navigation
  * `FocusNextNavEntry` / `FocusPreviousNavEntry` \- List navigation
  * `FocusNextRootNavEntry` / `FocusPreviousRootNavEntry` \- Section jumps


Sources: [crates/settings_ui/src/settings_ui.rs44-82](<https://github.com/zed-industries/zed/blob/4109c9dd/crates/settings_ui/src/settings_ui.rs#L44-L82>) [crates/settings_ui/src/settings_ui.rs712-721](<https://github.com/zed-industries/zed/blob/4109c9dd/crates/settings_ui/src/settings_ui.rs#L712-L721>) [crates/settings_ui/src/settings_ui.rs2286-2486](<https://github.com/zed-industries/zed/blob/4109c9dd/crates/settings_ui/src/settings_ui.rs#L2286-L2486>)

## Page Data Definition

Settings pages are defined declaratively in `page_data.rs`:


Each page definition specifies:

  * **Page title** \- Shown in navigation
  * **Section headers** \- Visual dividers within page
  * **Setting items** \- Individual configurable values 
    * Title and description text
    * Type-safe field accessor (`SettingField<T>`)
    * JSON path for deep linking
    * File scope (user/project/both)
    * Optional metadata (placeholder text, etc.)


The declarative structure makes it easy to add new settings by inserting items into the appropriate page.
