## Purpose and Scope

This document describes the Helix mode integration in Zed's Vim emulation system. Helix mode provides an alternative modal editing model inspired by the [Helix editor](<https://helix-editor.com/>), which differs from traditional Vim in its selection-first approach. For information about the broader Vim mode system and state management, see [Vim Mode](</zed-industries/zed/10-vim-mode>). For traditional Vim visual modes and selection handling, see [Visual Mode](</zed-industries/zed/10.3-visual-mode>).

Helix mode operates alongside traditional Vim mode, sharing the same infrastructure but with distinct behavior for motions, selections, and editing operations. Users can toggle between Vim and Helix modes using the `ToggleHelixMode` action.

**Sources** : [crates/vim/src/helix.rs1-83](<https://github.com/zed-industries/zed/blob/4109c9dd/crates/vim/src/helix.rs#L1-L83>) [crates/vim/src/vim.rs287-296](<https://github.com/zed-industries/zed/blob/4109c9dd/crates/vim/src/vim.rs#L287-L296>) [crates/vim/src/state.rs42-76](<https://github.com/zed-industries/zed/blob/4109c9dd/crates/vim/src/state.rs#L42-L76>)

## Core Differences from Traditional Vim

Helix mode fundamentally differs from Vim in its selection-centric philosophy:

Aspect| Vim Mode| Helix Mode  
---|---|---  
**Selection Model**|  Cursor-first; visual mode for selections| Selection-first; cursor always on a selection  
**Motion Behavior**|  Moves cursor; requires operator for action| Extends/modifies selections directly  
**Normal Mode**|  Cursor positioned between characters| Cursor positioned on a character (selection)  
**Visual Mode**|  Temporary mode entered with `v`| Primary mode (`HelixSelect`) for extended selections  
**Line Movements**| `j`/`k` move by buffer lines| `j`/`k` move by display lines by default  
  
In Helix mode, every cursor position represents a one-character selection, and motions extend these selections rather than just moving the cursor.

## Mode System

### Helix Modes

Helix mode introduces two distinct modes in the `Mode` enum:


**Diagram: Helix Mode State Machine**

**Mode Definitions** :

  * **`Mode::HelixNormal`** : The default Helix mode where each cursor is a one-character selection. Motions extend or move selections.
  * **`Mode::HelixSelect`** : Similar to Vim's visual mode, but selections are always visible and motions extend them. Entered by performing selection-extending motions.


The mode system recognizes Helix modes as visual-like through the `is_visual()` method, which returns `true` for `Mode::HelixSelect`.

### Mode Switching


**Diagram: Mode Initialization and Toggling**

The initialization logic ensures mutual exclusivity between Vim and Helix modes:

  * When Helix mode is enabled, `vim_mode` is set to `false` in settings
  * When Vim mode is enabled, `helix_mode` is set to `false` in settings
  * The initial mode is determined by `VimSettings::default_mode` (either `Normal` or `Insert`), which maps to `HelixNormal` or `Insert` when Helix mode is active

## Motion Handling

### Motion Execution Paths

Helix mode has distinct motion handling functions that differ from traditional Vim:


**Diagram: Helix Motion Execution Flow**

### Key Motion Differences

#### Word Motions

In Helix mode, word motions have different semantics:

  * **`Motion::NextWordStart`** : In Vim, moves to the first character of the next word. In Helix, moves to the character _before_ the next word (the last character of the current word/whitespace).
  * The implementation uses `helix_find_range_forward()` and `helix_find_range_backward()` to create selections based on boundary detection.


**Special Handling for NextWordStart** :

The `helix_select_motion()` function has special case logic for `NextWordStart` because Helix differs from Vim in this motion. It moves right, finds the boundary, then moves left to position correctly.

**Sources** : [crates/vim/src/helix.rs86-185](<https://github.com/zed-industries/zed/blob/4109c9dd/crates/vim/src/helix.rs#L86-L185>) [crates/vim/src/helix.rs315-337](<https://github.com/zed-industries/zed/blob/4109c9dd/crates/vim/src/helix.rs#L315-L337>)

#### Display Line vs Buffer Line

By default in Helix mode:

  * `j` and `k` move by display lines (visual lines accounting for wrapping)
  * `g j` and `g k` move by buffer lines (logical lines in the file)


This is inverted from traditional Vim, where `j`/`k` move by buffer lines and `g j`/`g k` move by display lines.

**Sources** : [assets/keymaps/vim.json418-425](<https://github.com/zed-industries/zed/blob/4109c9dd/assets/keymaps/vim.json#L418-L425>) [assets/keymaps/vim.json119-122](<https://github.com/zed-industries/zed/blob/4109c9dd/assets/keymaps/vim.json#L119-L122>)

### Boundary Detection

Helix uses specialized boundary detection functions:


**Diagram: Boundary Detection Logic**

The boundary functions check:

  1. Character kind transitions (word → punctuation, punctuation → whitespace, etc.)
  2. Newline characters (always boundaries)
  3. Whether we're moving into or out of whitespace

## Selection Management

### Selection Representation

In Helix mode, selections are always visible and extend inclusively:


**Diagram: Helix Selection Representation**

### Selection Adjustment Logic

When extending selections in `helix_select_motion()`:

  1. **Current head determination** : If the selection is not empty and not reversed, the current head is moved left by one character before motion calculation. This accounts for the fact that in forward selections, the cursor is just before the end.

  2. **Motion application** : The motion is applied to calculate a new head position.

  3. **End point adjustment** : If the selection is not reversed and not empty, the end point is moved right by one character to ensure the character under the cursor is included.

  4. **Anchor preservation** : When a selection reverses direction, the opposite end is adjusted to ensure the anchor character stays selected.


**Sources** : [crates/vim/src/helix.rs96-185](<https://github.com/zed-industries/zed/blob/4109c9dd/crates/vim/src/helix.rs#L96-L185>) [crates/vim/src/helix.rs287-313](<https://github.com/zed-industries/zed/blob/4109c9dd/crates/vim/src/helix.rs#L287-L313>)

`helix_move_and_collapse()`| Moves cursors and collapses to single points

## Helix-Specific Actions

### Core Actions


**Diagram: Helix-Specific Actions by Category**

### Selection Actions Implementation

**`HelixSelectLine`** : Extends each selection to encompass entire lines. Unlike Vim's `V` which enters a mode, this immediately extends the current selections.

**`HelixCollapseSelection`** : Collapses each multi-character selection to a single cursor position. If the selection is reversed, collapses to the head; otherwise, moves left from the head to collapse.

**`HelixKeepNewestSelection`** : Removes all selections except the one that was most recently created or modified. This is useful for quickly reducing multiple selections to one.

**Sources** : [crates/vim/src/helix.rs29-60](<https://github.com/zed-industries/zed/blob/4109c9dd/crates/vim/src/helix.rs#L29-L60>) [crates/vim/src/helix.rs62-83](<https://github.com/zed-industries/zed/blob/4109c9dd/crates/vim/src/helix.rs#L62-L83>) [crates/vim/src/normal.rs159-171](<https://github.com/zed-industries/zed/blob/4109c9dd/crates/vim/src/normal.rs#L159-L171>)

### Editing Actions Implementation

**`HelixYank`** : Yanks the current selections. Unlike Vim's yank which requires a motion or text object, Helix yank operates directly on the current selections.

**`HelixDelete`** : Deletes the current selections. Special handling ensures that if the selection is empty (just a cursor), it extends to the next character before deletion, then switches back to `HelixNormal` mode.

**`HelixSubstitute`** : Deletes the current selections and enters insert mode, similar to Vim's `c` but operating on selections directly.

**Sources** : [crates/vim/src/normal.rs144-157](<https://github.com/zed-industries/zed/blob/4109c9dd/crates/vim/src/normal.rs#L144-L157>) [crates/vim/src/helix/select.rs](<https://github.com/zed-industries/zed/blob/4109c9dd/crates/vim/src/helix/select.rs>)

### Integration Points with Core Vim

Helix mode reuses significant infrastructure from the Vim mode implementation:

  1. **Shared Mode Management** : Uses the same `Mode` enum and switching logic
  2. **Shared Editor Integration** : Both use `Vim::update_editor()` for editor operations
  3. **Shared Action System** : Both register actions through `Vim::action()`
  4. **Shared Register System** : Both use the same register system for yank/paste
  5. **Shared Settings** : Both use `VimSettings` with additional `HelixModeSetting`


The key difference is in how motions are handled and how selections are represented.

## Configuration and Settings

### Helix Mode Setting

Helix mode is controlled by the `HelixModeSetting`:


The setting is registered globally and can be toggled via:

  * Settings file: `"helix_mode": true` or `"helix_mode": false`
  * Command: `workspace::ToggleHelixMode` action


When toggled, the system ensures mutual exclusivity by disabling `vim_mode` when `helix_mode` is enabled and vice versa.

When Helix mode is enabled:

  * `default_mode: Normal` → initializes in `HelixNormal`
  * `default_mode: Insert` → initializes in `Insert`
